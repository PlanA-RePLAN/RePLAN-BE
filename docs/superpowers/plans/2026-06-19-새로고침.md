# 추천 새로고침(다시 추천받기) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 리플랜 추천과 목표추가 추천 API에 `refreshCount`(0~3) 한 필드를 더해, 같은 입력으로 회차별로 다른 결의 추천을 최대 3번 더 받을 수 있게 한다.

**Architecture:** 상태를 저장하지 않는다. 프론트가 0회차와 동일한 요청 몸체에 `refreshCount`만 증가시켜 다시 보내고, 백엔드는 그 숫자가 1~3이면 AI 프롬프트 맨 끝에 "회차별 스타일 블록"을 한 단락 덧붙여 추천을 생성한다. 0(또는 생략)이면 프롬프트는 기존과 100% 동일하다. 응답·저장 API는 변경 없음.

**Tech Stack:** Spring Boot 4.0.x, Java 17, JUnit5 + AssertJ. 빌드: `./gradlew -Dorg.gradle.java.home="$(/usr/libexec/java_home -v 17)" ...`

## Global Constraints

- `refreshCount` 허용 범위는 `0 <= refreshCount <= 3`. `null`은 0으로 취급. 범위 밖이면 `400`.
- 0회차(0 또는 null)일 때 프롬프트 문자열은 변경 전과 완전히 동일해야 한다(회귀 금지).
- 회차별 스타일 문구는 스펙 `docs/superpowers/specs/2026-06-19-새로고침-design.md` 3절 표를 그대로 따른다.
- 새 throw를 추가하면 해당 엔드포인트 `*ControllerDocs`의 `@ApiResponse` 에러 예시도 같이 갱신한다(프로젝트 규칙).
- 표/문서의 타입 표기는 JSON 타입 기준(`integer` 등), 설명·커밋은 쉬운 한국어. AI 작성 흔적 금지.
- 빌드 시 항상 `-Dorg.gradle.java.home="$(/usr/libexec/java_home -v 17)"` 를 붙인다.

```java
  /** 새로고침 횟수를 0~3로 검증하고 null은 0으로 정규화한다. */
  private int normalizeRefreshCount(Integer refreshCount) {
    if (refreshCount == null) {
      return 0;
    }
    if (refreshCount < 0 || refreshCount > 3) {
      throw new CustomException(ReplanErrorCode.REPLAN_INVALID_REFRESH_COUNT);
    }
    return refreshCount;
  }
```

- [ ] **Step 5: 서비스 검증 + 프롬프트 블록 구현**

`GoalAiService.java` `recommendTodos`에 검증을 추가하고, `buildRecommendPrompt`를 package-private으로 바꾸고 블록을 덧붙인다:

```java
  public TodoRecommendationResponse recommendTodos(TodoRecommendationRequest request) {
    int refreshCount = normalizeRefreshCount(request.refreshCount());
    String prompt = buildRecommendPrompt(request);
    String raw = callGemini(prompt);
    return parseRecommendResponse(raw);
  }

  /** 새로고침 횟수를 0~3로 검증하고 null은 0으로 정규화한다. */
  private int normalizeRefreshCount(Integer refreshCount) {
    if (refreshCount == null) {
      return 0;
    }
    if (refreshCount < 0 || refreshCount > 3) {
      throw new CustomException(GoalErrorCode.INVALID_REFRESH_COUNT);
    }
    return refreshCount;
  }
```

`buildRecommendPrompt` 시그니처에서 `private` 제거(package-private), 마지막 `return """...""".formatted(...);` 를 변수+블록으로 변경:

```java
  String buildRecommendPrompt(TodoRecommendationRequest req) {
    String deadlineInfo = buildDeadlineInfo(req.deadlineDate(), req.deadlineTime());
    String prompt =
        """
        ... (기존 프롬프트 본문 그대로) ...
        """
            .formatted(
                req.goal(),
                deadlineInfo,
                req.currentLevel() != null ? req.currentLevel() : "미입력",
                req.availableTime() != null ? req.availableTime() : "미입력",
                req.notes() != null ? req.notes() : "미입력");
    return prompt
        + refreshStyleBlock(req.refreshCount() == null ? 0 : req.refreshCount());
  }

  /** 새로고침 회차(1~3)에 맞는 스타일 안내 블록. 0/그 외는 빈 문자열(0회차는 프롬프트 변경 없음). */
  static String refreshStyleBlock(int refreshCount) {
    String line =
        switch (refreshCount) {
          case 1 ->
              "1회차(여유): 마감에 5~6일 버퍼를 넉넉히 둔다. 할 일은 아주 잘게(마이크로) 쪼개고, 쉬운 것부터 정순으로 배치한다.";
          case 2 ->
              "2회차(벼락치기): 버퍼 없이 혹은 마이너스로 빡빡하게 잡는다. 가장 핵심 1개(1-Pick) 위주로 줄이고, 어려운 것부터 역순으로 배치한다.";
          case 3 ->
              "3회차(환경 변경): 분량·난이도는 적정 수준으로 두되, 진행 시간대나 요일을 옮기는 방향으로 제안한다.";
          default -> null;
        };
    if (line == null) {
      return "";
    }
    return "\n\n[이번 새로고침 스타일]\n"
        + line
        + "\n위 스타일을 우선으로 적용하되, 결과는 위 공통 규칙(투두 형식·JSON 포맷)을 그대로 따른다.";
  }
```
