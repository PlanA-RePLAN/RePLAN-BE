# 추천 새로고침(다시 추천받기) 설계

**한 줄 목표:** 리플랜 추천과 목표추가 추천 화면에서 "새로고침" 버튼으로, 같은 입력에 대해 회차별로 다른 결의 추천을 최대 3번 더 받을 수 있게 한다.

**적용 범위:** 두 기능 모두
- 리플랜 추천 — `POST /api/replans/recommend`
- 목표추가 추천 — `POST /api/goals/ai/todo-recommendations`

---

## 1. 핵심 아이디어

지금은 추천을 누르면 AI가 계획 하나를 만들어 준다. 여기에 **새로고침**을 더해, 마음에 안 들면 다른 스타일의 계획을 **최대 3번 더** 받을 수 있게 한다.

새로고침은 **새 API도, DB 상태 저장도 없다.** 프론트가 기존 추천 요청에 `refreshCount`(0~3) 한 칸만 더 붙여서, **0회차 때 보낸 것과 똑같은 정보(앵커 ID·실패 이유·답변 등)를 그대로 다시 보낸다.** 백엔드는 그 숫자만 보고 AI 주문서(프롬프트)에 "이번엔 이 스타일로 짜줘"라는 회차 설명 블록을 끼워 넣는다.

---

## 2. 회차별 동작 규칙

| refreshCount | 동작 |
|---|---|
| `0` (또는 생략/null) | **지금과 100% 동일.** 실패 이유 로직 + 고정값을 그대로 적용한다. 프롬프트에 새 블록을 붙이지 않는다. |
| `1` · `2` · `3` | 프롬프트에 **회차별 스타일 블록**을 덧붙여 추천을 생성한다. |
| `4` 이상 / 음수 | `400` 에러로 막는다. |

### 고정값 처리 — 새로고침에서는 강제하지 않는다

문서에는 "실패 이유가 특정 변수를 고정하면(예: 체력 방전 → 시간대를 집중 시간으로 고정), 그 값은 어떤 템플릿에서도 유지" 규칙이 있다. 하지만 사용자가 **새로고침을 눌렀다는 것 자체가 "이건 별로니 다른 걸 달라"는 신호**이므로:

- **0차에서만** 고정값을 지킨다. (이미 배포된 현재 동작 그대로)
- **1·2·3차에서는** 고정값에 얽매이지 않고 회차 스타일대로 자유롭게 짠다.

→ 덕분에 "이유코드 → 고정축" 매핑표 같은 별도 코드가 필요 없다. 백엔드는 **회차별 스타일 문구만** 들고 있으면 된다.

### 이전 답변·입력은 그대로 흘러간다

프론트가 매 새로고침마다 0회차와 동일한 몸체를 다시 보내므로, 추가질문 답변(리플랜)이나 목표 정보(목표추가)는 자동으로 함께 전달된다. 새로고침 시에도 AI가 맥락으로 참고하되, 고정값으로 강제하지는 않는다.

---

## 3. 회차별 스타일 문구 (백엔드 상수)

문서 표를 그대로 옮긴다. `refreshCount > 0`일 때만 프롬프트 맨 끝에 아래 블록을 붙인다.

### 리플랜 추천 (문서 1번 표)

```text
[이번 새로고침 스타일 — N회차]
1회차(여유): 기간은 기존 예상 소요의 1.5배 이상으로 넉넉히. 할 일은 아주 잘게(마이크로) 쪼개고, 가장 쉽고 만만한 것부터 시작하도록 정순으로 배치한다.
2회차(벼락치기): 버퍼 없이 짧게(15~25분 타임박싱). 핵심 1개(1-Pick)만 남기고, 가장 어렵고 중요한 것부터 처리하도록 역순으로 배치한다.
3회차(환경 제약): 기간·덩어리·난이도는 적정 수준으로 두되, 시간대를 옮기거나(다른 시간/요일) 장소·환경을 통제하는(조용한 곳 이동, 방해금지 등) 방향으로 제안한다.

위 스타일을 우선으로 적용하되, 결과는 기존 공통 규칙(투두 형식·action 종류·JSON 포맷)을 그대로 따른다.
```

### 목표추가 추천 (문서 2번 표)

```text
[이번 새로고침 스타일 — N회차]
1회차(여유): 마감에 5~6일 버퍼를 넉넉히 둔다. 할 일은 아주 잘게(마이크로) 쪼개고, 쉬운 것부터 정순으로 배치한다.
2회차(벼락치기): 버퍼 없이 혹은 마이너스로 빡빡하게. 가장 핵심 1개(1-Pick) 위주로 줄이고, 어려운 것부터 역순으로 배치한다.
3회차(환경 변경): 분량·난이도는 적정 수준으로 두되, 진행 시간대나 요일을 옮기는 방향으로 제안한다.

위 스타일을 우선으로 적용하되, 결과는 기존 공통 규칙(투두 형식·JSON 포맷)을 그대로 따른다.
```

> N회차의 N은 실제 `refreshCount` 값으로 치환하고, 해당 회차 한 줄만 넣어도 되고 전체를 넣고 "이번은 N회차" 를 명시해도 된다. 구현 단계에서 한 회차 줄만 선택해 넣는 방식으로 한다(프롬프트를 짧게 유지).

---

## 4. API 변경

### 리플랜 — `ReplanRecommendRequest`

`refreshCount` 필드 추가 (선택, 기본 0):

```java
@Schema(description = "새로고침 횟수(0~3). 0 또는 생략은 첫 추천, 1~3은 다시 추천", example = "0")
Integer refreshCount
```

- `ReplanService.recommend()` 에서 검증 후 `RecommendInput`/프롬프트 빌더로 전달.
- `RecommendInput`에 `int refreshCount` 추가.
- `ReplanAiService.buildRecommendPrompt()`에서 `refreshCount > 0`이면 위 리플랜 스타일 블록을 덧붙인다.

### 목표추가 — `TodoRecommendationRequest`

동일하게 `refreshCount`(선택, 기본 0) 추가:

- `GoalAiService.recommendTodos()` → `buildRecommendPrompt()`에서 `refreshCount > 0`이면 목표추가 스타일 블록을 덧붙인다.

### 검증

- 두 곳 모두 `refreshCount`가 `null`이면 0으로 처리.
- `0 <= refreshCount <= 3` 범위를 벗어나면 `400`.
  - 리플랜: 새 에러코드 `REPLAN_INVALID_REFRESH_COUNT` 추가.
  - 목표추가: 새 에러코드 `INVALID_REFRESH_COUNT`(GoalErrorCode) 추가.

---

## 5. 응답

응답 형태는 **변경 없음.** 회차와 무관하게 기존과 동일한 추천 결과(operations / todos)를 그대로 내려준다. 프론트는 받은 결과를 화면에 다시 그리기만 하면 된다.

---

## 6. 짝꿍 동기화 (프로젝트 규칙)

- 새 throw(`REPLAN_INVALID_REFRESH_COUNT`, `INVALID_REFRESH_COUNT`)를 추가하므로, 해당 엔드포인트의 `*ControllerDocs` `@ApiResponse` 에러 예시도 같이 추가한다.
- 두 Request DTO에 새 필드 `@Schema` 작성, Docs의 Request Body 표에도 `refreshCount` 행 추가(선택 필드 주석 포함).

---

## 7. 테스트

- 리플랜: `refreshCount`가 0/생략이면 프롬프트에 스타일 블록이 없고, 1~3이면 해당 회차 블록이 포함된다(프롬프트 문자열 검증).
- 리플랜: `refreshCount`가 4 또는 음수면 `REPLAN_INVALID_REFRESH_COUNT`로 막힌다.
- 목표추가: 동일한 두 테스트(GoalErrorCode).
- 0차 동작이 기존과 동일함(회귀)을 확인한다.

---

## 8. 범위에서 제외 (YAGNI)

- 새로고침 횟수의 서버측 영속 저장 — 하지 않음(프론트가 횟수 전달).
- 이유코드 → 고정축 매핑표 — 하지 않음(새로고침은 고정값 비강제).
- 응답/저장 API 변경 — 없음.
